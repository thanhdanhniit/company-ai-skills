# Code Quality Standards

This skill ensures that all generated code follows modern software engineering best practices.

Goals:

* maintainable codebases
* clear architecture
* readable and self-documenting code
* minimal technical debt
* easy long-term evolution

These standards must be applied whenever generating or modifying code.

---

# Core Engineering Principles

## 1. Single Responsibility Principle

Every module, class, and function must have **one clear responsibility**.

Bad example:

```
function processOrder(order) {
  validateOrder(order)
  chargeCreditCard(order)
  updateDatabase(order)
  sendEmail(order)
}
```

Better design:

```
validateOrder(order)
processPayment(order)
saveOrder(order)
sendOrderConfirmation(order)
```

Benefits:

* easier testing
* simpler maintenance
* clearer logic boundaries

---

## 2. Prefer Small Functions

Functions should generally be **less than 30 lines**.

Large functions are difficult to understand and maintain.

If a function grows too large:

* extract helper functions
* separate business logic
* move reusable logic to utilities

---

## 3. Avoid Deep Nesting

Maximum recommended nesting depth: **3 levels**

Bad:

```
if (user) {
  if (user.active) {
    if (user.role === "admin") {
      performAction()
    }
  }
}
```

Better:

```
if (!user) return
if (!user.active) return
if (user.role !== "admin") return

performAction()
```

Use **guard clauses** and **early returns**.

---

## 4. Meaningful Naming

Names must clearly communicate intent.

Good examples:

```
createIncidentTicket
fetchUserProfile
calculateInvoiceTotal
validatePaymentRequest
```

Bad examples:

```
doStuff
handleThing
processData
tempFunction
```

Naming rules:

* use verbs for functions
* use nouns for objects
* avoid abbreviations
* prefer clarity over brevity

---

## 5. Avoid Code Duplication

Follow the **DRY principle (Don't Repeat Yourself)**.

If logic appears multiple times:

* extract shared functions
* create reusable utilities
* centralize business logic

Benefits:

* fewer bugs
* easier updates
* better consistency

---

## 6. Prefer Composition Over Inheritance

Avoid deep inheritance hierarchies.

Bad design:

```
BaseService
  └ UserService
       └ AdminService
```

Better approach:

```
UserService
PermissionService
NotificationService
```

Prefer:

* modular services
* functional composition
* dependency injection

---

## 7. Avoid Magic Numbers

Never embed unexplained numeric values in code.

Bad:

```
if (retryCount > 5)
```

Better:

```
const MAX_RETRY_ATTEMPTS = 5

if (retryCount > MAX_RETRY_ATTEMPTS)
```

Benefits:

* readability
* easier configuration
* easier updates

---

## 8. Consistent Code Formatting

Generated code must follow consistent formatting rules.

Recommended tools:

JavaScript / TypeScript

* ESLint
* Prettier

Java

* Checkstyle
* Spotless

Python

* Black
* Flake8

Formatting should always be automated.

---

## 9. Self-Documenting Code

Code should be readable without excessive comments.

Use comments only for:

* complex algorithms
* architectural decisions
* non-obvious business rules

Avoid redundant comments.

Bad:

```
# increment i
i = i + 1
```

---

# Error Handling

Code must implement proper error handling.

Principles:

* never silently ignore errors
* return meaningful error messages
* log unexpected failures

Example:

```
try {
  processPayment()
} catch (error) {
  logger.error("Payment processing failed", error)
  throw new PaymentProcessingError()
}
```

---

# Logging Standards

Use structured logging.

Bad:

```
console.log("user created")
```

Better:

```
logger.info("User created", { userId })
```

Logging should include:

* event name
* relevant identifiers
* contextual metadata

Never log sensitive information.

---

# Dependency Management

Dependencies should be minimal and well-maintained.

Rules:

* avoid unnecessary libraries
* prefer widely adopted libraries
* keep dependencies updated

---

# Performance Awareness

Generated code should consider performance implications.

Guidelines:

* avoid unnecessary loops
* avoid repeated expensive operations
* cache frequently used results when appropriate
* avoid blocking operations in asynchronous environments

---

# Testability

Code should be designed to be easily testable.

Best practices:

* separate business logic from infrastructure
* use dependency injection
* avoid tight coupling
* keep functions deterministic when possible

---

# Output Requirements

Generated code must be:

* modular
* readable
* maintainable
* testable
* scalable
* consistent across the project

Code that does not follow these principles must be refactored before final output.
