# Security Best Practices

This skill ensures that all generated code follows secure development practices to prevent common application vulnerabilities.

Security must be considered during:

* system design
* API development
* data storage
* authentication
* authorization
* logging
* deployment

The goal is to minimize security risks and protect sensitive data.

---

# Core Security Principles

## 1. Principle of Least Privilege

Systems, services, and users must be granted **only the minimum permissions required** to perform their tasks.

Example:

Bad:

* database user with full admin privileges
* services running as root

Better:

* read-only roles for read operations
* scoped IAM roles or service accounts
* restricted database permissions

Benefits:

* reduces attack surface
* limits damage from compromised components

---

## 2. Secure Authentication

Authentication mechanisms must follow modern security standards.

Best practices:

* never store plaintext passwords
* enforce strong password policies
* support multi-factor authentication (MFA) where applicable
* use secure token-based authentication

Password storage must use hashing algorithms such as:

* bcrypt
* Argon2
* PBKDF2

Never implement custom cryptography.

---

## 3. Secure Authorization

Authorization must always be enforced **on the server side**.

Never rely solely on frontend validation.

Example:

Bad:

```
if (user.role === "admin") {
  showAdminButton()
}
```

Better:

Server validates authorization before executing protected operations.

Authorization models may include:

* Role-Based Access Control (RBAC)
* Attribute-Based Access Control (ABAC)

---

## 4. Input Validation

All external input must be validated.

Sources of external input include:

* API requests
* form submissions
* query parameters
* headers
* uploaded files

Best practices:

* validate type
* validate format
* validate allowed values
* enforce length limits

Example:

```
validateEmailFormat(email)
validateStringLength(name, 1, 100)
```

Never trust client input.

---

## 5. Prevent Injection Attacks

Injection vulnerabilities are among the most critical risks.

Applications must prevent:

* SQL injection
* NoSQL injection
* command injection
* LDAP injection

Best practices:

* always use parameterized queries
* never concatenate user input into database queries
* use ORM frameworks where appropriate

Bad:

```
"SELECT * FROM users WHERE email = '" + email + "'"
```

Better:

```
SELECT * FROM users WHERE email = ?
```

---

## 6. Protect Against Cross-Site Scripting (XSS)

Applications must prevent execution of malicious scripts in browsers.

Best practices:

* escape all user-generated content
* sanitize HTML inputs
* use trusted libraries for sanitization
* avoid dangerously injecting raw HTML

Example protection methods:

* HTML escaping
* Content Security Policy (CSP)

---

## 7. Prevent Cross-Site Request Forgery (CSRF)

State-changing operations must be protected from CSRF attacks.

Mitigation strategies:

* CSRF tokens
* SameSite cookies
* origin validation

Applicable to:

* form submissions
* authenticated API requests

---

## 8. Secure API Design

APIs must enforce security controls.

Best practices:

* require authentication for protected endpoints
* implement authorization checks
* apply request rate limiting
* validate all request payloads

Sensitive endpoints should include:

* audit logging
* abuse detection
* throttling protection

---

## 9. Secure Data Storage

Sensitive data must be protected.

Never store sensitive data in plaintext.

Examples of sensitive data:

* passwords
* access tokens
* personal data
* financial information

Protection strategies:

* encryption at rest
* secure key management
* restricted database access

Sensitive secrets should never be stored in source code.

Use:

* environment variables
* secret management systems

---

## 10. Secure Data Transmission

All network communication must be encrypted.

Best practices:

* enforce HTTPS
* disable insecure protocols
* validate TLS certificates

Never send sensitive data through unsecured channels.

---

## 11. Secure Logging

Logs must never expose sensitive information.

Do not log:

* passwords
* access tokens
* session IDs
* personal identifiable information

Bad:

```
logger.info("User login", { password })
```

Better:

```
logger.info("User login attempt", { userId })
```

Logs should include:

* timestamp
* event type
* request identifier
* user identifier (if applicable)

---

## 12. Dependency Security

Third-party dependencies introduce security risks.

Best practices:

* keep dependencies updated
* remove unused libraries
* scan for vulnerabilities

Recommended tools:

* dependency scanners
* vulnerability databases
* automated security alerts

---

## 13. Secure Error Handling

Error messages must not reveal internal system details.

Bad:

```
Database connection failed at db.internal.server:5432
```

Better:

```
Internal server error
```

Detailed errors should be logged internally but hidden from end users.

---

## 14. Rate Limiting and Abuse Protection

APIs should implement protections against abuse.

Best practices:

* request rate limiting
* API throttling
* bot detection
* account lockout policies

These mechanisms prevent:

* brute force attacks
* denial-of-service attempts

---

## 15. Secure File Upload Handling

If file uploads are supported, apply strict validation.

Best practices:

* restrict file types
* enforce size limits
* scan uploaded files for malware
* store files outside executable directories

Never directly execute uploaded files.

---

# Secure Development Workflow

Security must be integrated throughout the development lifecycle.

Recommended practices:

* secure code reviews
* automated security scanning
* vulnerability testing
* dependency monitoring

Security should be treated as a **first-class engineering requirement**, not an afterthought.

---

# Output Requirements

All generated code must follow these security standards.

The code must:

* validate external input
* enforce authentication and authorization
* protect sensitive data
* avoid insecure cryptographic practices
* prevent common web vulnerabilities
* follow secure logging practices

If a generated implementation violates these rules, it must be corrected before final output.
