# Testing Standards

This skill ensures that generated software includes proper automated testing practices.

Testing is required to ensure:

* correctness
* stability
* maintainability
* safe refactoring
* regression prevention

Every production system must include automated tests.

Testing must be considered part of the development process and not an optional activity.

---

# Testing Principles

## 1. Test Pyramid

Testing strategy must follow the **test pyramid model**.

Test distribution should prioritize:

1. Unit Tests (majority)
2. Integration Tests
3. End-to-End Tests (minimal)

Example structure:

```
       E2E Tests
     Integration Tests
        Unit Tests
```

Benefits:

* fast feedback
* reliable builds
* easier debugging

---

## 2. Automated Testing Only

All tests must be automated.

Avoid reliance on manual testing for validation of application logic.

Automated tests should run:

* during local development
* during CI pipelines
* before deployment

---

## 3. Unit Testing

Unit tests verify **individual functions or modules**.

Characteristics:

* fast
* isolated
* deterministic
* no external dependencies

Unit tests must:

* test business logic
* validate edge cases
* verify expected outputs

Example test targets:

* validation logic
* data transformation
* business rules
* utility functions

Unit tests should **not**:

* connect to databases
* call external APIs
* depend on network services

External dependencies should be mocked.

---

## 4. Integration Testing

Integration tests verify that **multiple components work together correctly**.

Examples:

* API endpoints interacting with database
* service layer calling repositories
* event processing pipelines

Integration tests may include:

* database connections
* message queues
* service integrations

However, tests should still run in controlled environments.

---

## 5. End-to-End Testing

End-to-end (E2E) tests simulate **real user workflows**.

These tests validate:

* system integration
* user interaction flows
* full application behavior

Examples:

* user login workflow
* incident creation workflow
* ticket escalation process

E2E tests should be limited due to:

* slower execution
* higher maintenance cost

---

# Test Coverage

Code coverage must be monitored to ensure critical functionality is tested.

Recommended coverage targets:

* Business logic: 80% or higher
* Critical modules: 90% or higher
* Overall project coverage: at least 70%

Coverage should focus on meaningful tests rather than superficial metrics.

---

# Test Structure

Tests must follow a clear structure.

Preferred pattern: **Arrange, Act, Assert**

Example:

```
Arrange
  create test input data

Act
  execute function

Assert
  verify expected result
```

Example pseudocode:

```
test("should calculate total price") {
  Arrange:
    items = [10, 20]

  Act:
    total = calculateTotal(items)

  Assert:
    expect(total).toEqual(30)
}
```

---

# Deterministic Tests

Tests must produce consistent results every time they run.

Avoid:

* reliance on system time
* random values
* external services
* shared mutable state

If randomness is required, use deterministic seeds.

---

# Test Data Management

Test data should be:

* minimal
* isolated
* predictable

Avoid using production data.

Test data must not contain sensitive information.

Use fixtures or factories to generate consistent test data.

---

# Mocking and Stubbing

External dependencies should be mocked during unit testing.

Examples:

* HTTP APIs
* databases
* message queues
* third-party services

Benefits:

* faster tests
* deterministic behavior
* isolated logic validation

Mocks should only simulate behavior necessary for the test.

---

# Continuous Integration Testing

All automated tests must run in CI pipelines.

CI pipelines should:

* run tests on every commit
* block merges on failed tests
* generate test reports
* report coverage metrics

This ensures early detection of defects.

---

# Regression Testing

When fixing a bug:

1. write a failing test reproducing the bug
2. fix the bug
3. ensure the test passes

This ensures the issue cannot reappear.

---

# Naming Conventions

Test names must clearly describe expected behavior.

Good examples:

```
shouldCreateIncidentWhenValidDataProvided
shouldRejectInvalidEmailFormat
shouldEscalateTicketAfterTimeout
```

Bad examples:

```
test1
testFunction
validateStuff
```

Tests should read like documentation.

---

# Performance Awareness

Tests should execute quickly.

Slow tests reduce developer productivity.

Guidelines:

* unit tests should run in milliseconds
* integration tests should run in seconds
* avoid unnecessary setup or teardown steps

---

# Test Isolation

Each test must run independently.

Tests must not depend on:

* execution order
* shared global state
* previous test results

Test frameworks should reset state between tests.

---

# Failure Diagnostics

Test failures must provide useful information.

Error messages should clearly explain:

* expected result
* actual result
* context of failure

Example:

```
Expected total price to equal 30
Received 25
```

Clear failures accelerate debugging.

---

# Output Requirements

Generated code must include appropriate automated tests.

Tests must:

* validate business logic
* cover critical paths
* run reliably in CI environments
* follow the testing pyramid model

Testing is considered a required component of high-quality software.
